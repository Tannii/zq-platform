### 1. 项目概述

本项目是基于Python + Django + Django REST Framework（DRF）构建的通用后端服务框架，旨在提供标准化、高可扩展、易维护的后端支撑能力，可快速适配各类业务场景（如后台管理系统、小程序后端、APP后端、PC端应用后端等）。项目核心聚焦于基础能力封装与业务模块解耦，内置用户认证、权限管理、数据校验、文件上传、定时任务调度等通用功能，减少重复开发工作量，同时支持业务模块灵活扩展，满足不同项目的个性化需求。

项目整体采用分层架构设计，严格遵循RESTful设计规范，兼顾开发效率与系统稳定性，适用于中小规模项目快速落地，也可通过扩展适配大型项目的高并发、高可用需求。

### 2. 核心技术栈与环境要求

#### 2.1 核心技术栈

项目技术选型以“稳定、高效、易用”为核心，选用成熟的技术框架与工具，具体如下：

| 技术/框架 | 版本建议 | 核心作用 |

| -------- | -------- | -------- |

| Python | 3.8+ | 项目开发主语言，提供简洁、高效的代码编写体验 |

| Django | 4.2+ | 后端Web开发框架，封装了ORM、路由、模板等核心能力，提升开发效率 |

| Django REST Framework | 3.14+ | RESTful接口开发框架，提供序列化、视图、权限、分页等标准化组件 |

| djangorestframework-simplejwt | 5.2+ | JWT身份认证组件，实现用户登录、Token刷新、权限校验等功能 |

| Celery/APScheduler | 5.2+/3.10+ | 定时任务调度工具，支持周期任务、延迟任务，如数据同步、报表生成、定时推送等 |

| MySQL | 8.0+ | 关系型数据库，用于存储项目核心业务数据，支持事务、索引等特性 |

| Redis | 6.0+ | 缓存数据库，用于存储Token、热点数据、任务队列，提升系统响应速度 |

| Pillow | 9.0+ | 图片处理工具，支持图片压缩、格式转换等，配合文件上传功能使用 |

| loguru | 0.7+ | 日志管理工具，替代Django默认日志，支持日志分级、滚动存储，便于问题排查 |

| django-cors-headers | 4.0+ | 跨域请求处理组件，解决前后端分离架构下的跨域问题 |

#### 2.2 环境要求

项目可运行于多种操作系统，需满足以下硬件与软件环境要求，确保系统稳定运行：

| 环境类型 | 具体要求 |

| -------- | -------- |

| 操作系统 | Linux（CentOS 7+/Ubuntu 20.04+，推荐生产环境）、Windows 10/11（开发环境）、macOS 12+（开发环境） |

| 依赖管理工具 | pip 22.0+ 或 poetry 1.4+，用于管理项目依赖包 |

| 网络要求 | 开放指定端口（默认8000，可配置），确保数据库、Redis服务可正常访问，前后端可正常通信 |

| 硬件要求（最低） | CPU：2核，内存：4G，磁盘：20G；生产环境建议CPU：4核+，内存：8G+，磁盘：50G+ |

| 其他 | 开发环境需安装Git（版本控制）、IDE（如PyCharm），生产环境需配置Nginx（反向代理）、Supervisor（进程管理） |

### 4. 项目核心功能模块说明

项目按“通用基础+业务扩展”的思路设计，核心模块分为通用模块和业务模块，各模块职责清晰、相互独立，同时支持灵活扩展。

#### 4.1 通用基础模块

通用基础模块是项目的核心支撑，提供所有业务模块共用的能力，无需重复开发，具体如下：

| 模块名称 | 核心功能 | 具体说明 |

| -------- | -------- | -------- |

| 认证模块（基于JWT） | 用户登录、Token刷新、退出登录、权限校验 | 支持用户名/手机号/邮箱登录，生成访问Token和刷新Token，Token过期自动失效，支持权限分级校验 |

| 通用工具模块 | 工具函数、常量、校验、分页、过滤 | 提供加密（MD5、SHA256）、时间格式化、数据校验、文件操作等工具函数，统一常量和分页/过滤规则 |

| 文件上传模块 | 图片/文档上传、文件存储、文件访问 | 支持多类型文件上传，自动按日期分类存储，返回文件访问URL，支持文件大小、格式限制 |

| 定时任务模块 | 任务定义、调度配置、任务执行 | 支持周期任务（如每日凌晨数据统计）、延迟任务（如10分钟后发送通知），可配置任务优先级和重试机制 |

| 全局异常处理模块 | 异常捕获、错误响应标准化 | 捕获项目中所有未处理的异常（如数据库异常、参数错误、权限异常），返回统一格式的错误响应，便于前后端对接 |

| 日志模块 | 日志记录、分级存储、问题排查 | 按系统、业务、错误分级记录日志，支持日志滚动存储，包含请求参数、响应结果、错误堆栈等信息，便于问题定位 |

#### 4.2 业务模块（示例）

业务模块基于通用基础模块扩展，按业务域拆分，可根据实际项目需求新增、修改或删除，以下为示例模块：

| 模块名称 | 核心功能 | 扩展说明 |

| -------- | -------- | -------- |

| 用户模块 | 用户注册、登录、个人信息管理、密码重置 | 可扩展手机号验证码登录、第三方登录（微信、QQ）、用户等级、积分等功能 |

| 权限模块 | 角色管理、菜单管理、接口权限分配 | 支持RBAC权限模型，可按角色分配不同接口访问权限，适配后台管理系统的权限管控需求 |

| 订单模块（示例） | 订单创建、查询、更新、删除、订单统计 | 可替换为商品模块、文章模块、表单模块等，适配具体业务场景 |

### 5. 项目部署与运行指南

本章节详细说明项目的环境搭建、配置修改、数据库初始化、项目启动及部署流程，适用于开发、测试、生产环境的部署与运行。

#### 5.1 环境搭建（开发环境）

开发环境用于日常开发、调试代码，步骤如下：

1. 克隆项目代码：使用Git克隆项目仓库到本地，命令如下：

git clone <项目仓库地址>
cd backend-django  # 进入项目根目录

2. 创建并激活虚拟环境：为避免依赖冲突，建议使用虚拟环境隔离项目依赖，命令如下：

# Linux/macOS系统
python -m venv env  # 创建虚拟环境
source env/bin/activate  # 激活虚拟环境

# Windows系统
python -m venv env
env\Scripts\activate

激活成功后，终端会显示“(env)”标识，表示当前处于虚拟环境中。

3. 安装依赖包：使用pip安装项目所需的所有依赖，命令如下：

pip install -r requirements.txt

若安装失败，可尝试升级pip或指定依赖包版本。

#### 5.2 配置修改

项目配置按环境拆分，开发环境需修改以下核心配置（生产环境配置类似，需适配线上环境）：

1. 数据库配置：修改core/settings/dev.py中的数据库配置，适配本地MySQL环境：

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.mysql',  # 数据库引擎
        'NAME': 'django_db',                   # 数据库名称（需提前创建）
        'USER': 'root',                        # 数据库账号
        'PASSWORD': '123456',                  # 数据库密码
        'HOST': '127.0.0.1',                   # 数据库地址（本地为127.0.0.1）
        'PORT': '3306',                        # 数据库端口（默认3306）
        'OPTIONS': {
            'charset': 'utf8mb4'               # 字符集，支持emoji等特殊字符
        }
    }
}

2. Redis配置：修改core/settings/dev.py中的Redis配置，用于缓存、Token存储和定时任务队列：

CACHES = {
    'default': {
        'BACKEND': 'django_redis.cache.RedisCache',
        'LOCATION': 'redis://127.0.0.1:6379/1',  # Redis地址+数据库编号（1表示第1个数据库）
        'OPTIONS': {
            'CLIENT_CLASS': 'django_redis.client.DefaultClient',
            'PASSWORD': ''  # 若Redis设置了密码，填写对应密码
        }
    }
}

# JWT配置（可选，可调整Token有效期）
SIMPLE_JWT = {
    'ACCESS_TOKEN_LIFETIME': timedelta(hours=2),  # 访问Token有效期2小时
    'REFRESH_TOKEN_LIFETIME': timedelta(days=7),  # 刷新Token有效期7天
}

3. 其他配置：可根据需求修改端口、日志路径、媒体文件存储路径等配置，均在core/settings/dev.py中。

#### 5.3 数据库初始化

数据库初始化用于创建数据库表结构、导入基础数据，步骤如下：

1. 创建数据库：打开MySQL客户端，创建项目所需的数据库（名称与配置文件中一致）：

CREATE DATABASE django_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

2. 执行数据库迁移：Django通过迁移文件创建数据库表结构，命令如下：

python manage.py makemigrations  # 生成迁移文件（检测模型变化）
python manage.py migrate         # 执行迁移文件，创建表结构

3. 导入初始化数据（可选）：若项目有基础初始化数据（如默认超级管理员、字典数据），执行以下命令导入：

python manage.py loaddata db_init.json

4. 创建超级管理员：用于登录Django后台管理页面，管理数据，命令如下：

python manage.py createsuperuser

执行命令后，按提示输入用户名、邮箱、密码即可创建超级管理员。

#### 5.4 项目启动

开发环境启动项目分为Web服务启动和定时任务启动，步骤如下：

1. 启动Web服务：

python manage.py runserver 0.0.0.0:8000

启动成功后，可通过http://127.0.0.1:8000访问项目，接口根路径为http://127.0.0.1:8000/api/v1/，后台管理页面为http://127.0.0.1:8000/admin/。

2. 启动定时任务（若有）：

python start_scheduler.py

启动成功后，定时任务会按配置的周期自动执行。

#### 5.5 生产环境部署（简要说明）

生产环境部署需考虑稳定性、安全性和性能，步骤如下：

1. 环境准备：服务器安装Python、MySQL、Redis、Nginx、Supervisor等软件，配置好环境变量。

2. 代码部署：将项目代码上传到服务器，创建虚拟环境，安装依赖，修改生产环境配置（core/settings/prod.py）。

3. 数据库配置：配置线上MySQL、Redis，执行数据库迁移，导入初始化数据。

4. 服务部署：使用Gunicorn作为Web服务器（替代Django自带的开发服务器），配置Nginx反向代理，使用Supervisor管理Web服务和定时任务进程。

5. 安全配置：开启HTTPS，关闭调试模式，配置防火墙，限制数据库、Redis访问权限。

### 6. 项目维护与问题排查

#### 6.1 日常维护

- 依赖维护：定期更新项目依赖包，修复安全漏洞，执行pip list --outdated查看可更新依赖，执行pip install --upgrade 依赖包名更新。

- 日志维护：定期清理日志文件，避免磁盘空间不足，可配置日志滚动存储（loguru已默认支持）。

- 数据维护：定期备份数据库，避免数据丢失，可配置定时备份任务（通过定时任务模块实现）。

- 进程维护：生产环境通过Supervisor监控Web服务和定时任务进程，若进程异常退出，自动重启。

#### 6.2 问题排查

项目运行中出现问题，可按以下步骤排查：

1. 查看日志：优先查看logs/error.log错误日志，获取错误堆栈信息，定位错误原因。

2. 检查配置：确认数据库、Redis配置是否正确，是否能正常连接；检查接口路径、请求参数是否符合约定。

3. 调试代码：开发环境可开启调试模式（DEBUG=True），查看详细的错误信息和请求流程；生产环境可通过打印日志的方式调试。

4. 检查依赖：确认依赖包版本是否兼容，是否存在缺失的依赖包。

### 7. 项目扩展说明

项目设计时考虑了扩展性，可根据实际业务需求灵活扩展功能，扩展规则如下：

1. 新增业务模块：在application/目录下新建模块（如goods/商品模块），按“models.py → serializers.py → views.py → urls.py”的结构编写代码，最后在项目主路由中注册该模块的路由。

2. 扩展通用功能：若需新增通用工具、中间件等，可在common/或core/目录下新增对应文件，确保代码可复用。

3. 接口扩展：新增接口需遵循RESTful设计规范，保持请求方式、响应格式与通用约定一致，同时补充接口文档。

4. 技术栈扩展：可根据需求新增技术组件，如引入Elasticsearch实现全文检索、引入RabbitMQ实现消息队列等，需确保与现有技术栈兼容。

### 8. 注意事项

- 开发环境与生产环境配置严格区分，生产环境需关闭DEBUG模式，避免泄露敏感信息。

- 用户密码需加密存储，项目已默认使用Django自带的密码加密机制，禁止明文存储密码。

- 生产环境建议开启HTTPS，避免Token、敏感数据在传输过程中泄露。

- 定时任务的执行周期和逻辑需谨慎配置，避免占用过多服务器资源。

- 数据库操作需做好异常捕获，避免因数据异常导致系统崩溃；重要数据需定期备份。

- 新增代码需编写测试用例，确保代码质量；提交代码前需进行代码审核，避免引入bug。
